# Feature Specification: Custom Events

**Feature Branch**: `004-custom-events`

**Created**: 2026-05-18

**Status**: Draft

**Input**: User description: "Slice 004 — Custom Events. Analyzer's first in-request capture path beyond pageviews. Adds a client-callable API (`analyzer.send("event", category, action, label?, value?)`) that posts to an Analyzer-owned management endpoint; the endpoint resolves the visitor's active session (slice 003), persists one row in a new `analyzerCustomEvent` table, and lights up `IAnalyticsEventStateProvider.CurrentRequestCustomEvents` for in-request consumers. Adds a hard-delete cascade step on operator anonymisation. Codifies validation + RBAC + audit + anti-forgery on the new management surface."

## Clarifications

### Session 2026-05-18

- Q: Should `analyzer.send(...)` calls advance the session's `lastActivityUtc` (keep the session alive on in-page engagement) without incrementing `pageviewCount`? → A: Yes — add a new `IAnalyzerSessionRepository.TouchAsync(sessionKey, lastActivityUtc)` path that issues 1 indexed UPDATE with no `pageviewCount` change. Custom events keep the session warm; matches the Mixpanel/GA "engagement keeps session alive" pattern and avoids the surprising long-lived-SPA-session-times-out case.
- Q: What does `analyzer.send(...)` return on the JS side? → A: `Promise<{ eventKey: string }>` — resolves on HTTP 202 with the server-issued `eventKey`; rejects on 4xx/5xx with `{ status: number, message: string }`. Aligns with `fetch` semantics and lets future SPA consumers correlate the event for follow-up reads.

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Page scripts record custom engagement events (Priority: P1)

When an authenticated EntraID employee's page script invokes `analyzer.send("event", category, action, label?, value?)`, the call posts the event payload to Analyzer's management endpoint, which authenticates the request via the standard Umbraco backoffice session, resolves the visitor's active `(visitorProfileKey, deviceKey)` session via slice-003's `IAnalyzerSessionResolver` (in-request — first reliable consumer of `IAnalyticsEventStateProvider.CurrentSession`), and writes a single `analyzerCustomEvent` row keyed to that session. The new row carries `(sessionKey, visitorProfileKey, receiptKey?, category, action, label?, value?, receivedUtc)`. Slice-002's `IAnalyticsEventStateProvider` gains an additive read-only member `CurrentRequestCustomEvents` exposing the in-flight event(s) to other in-process consumers during the same request scope.

**Why this priority**: Custom events are Analyzer's first in-request capture path — every later slice (video, forms, scroll, search, content-app drill-downs, reports) layers on top of this surface. The slice also validates two slice-003-design assumptions concretely: (a) the session resolver is callable from a synchronous in-request code path and produces a reliable session reference, (b) `IAnalyticsEventStateProvider.CurrentSession` populates reliably for in-request consumers (vs the rare-population caveat on the fire-and-forget pageview path documented in slice 002 + slice 003). P1 because every operator-defined engagement metric the future reports surface depends on this path working.

**Independent Test**: From a Razor-rendered page in the host Umbraco site, run `analyzer.send("event", "engagement", "click", "header-cta")` in the browser console. Read `analyzerCustomEvent` afterwards: exactly one row with `category = "engagement"`, `action = "click"`, `label = "header-cta"`, `value IS NULL`, `sessionKey` matching the visitor's active session, `visitorProfileKey` matching the resolved EntraID identity, `receivedUtc` within 1 second of the call, and `receiptKey` either null OR populated if the request happened to be the same one that captured the pageview (rare; mostly null because the page script's fetch is a separate request from the page render).

**Acceptance Scenarios**:

1. **Given** an authenticated EntraID employee with an active in-progress session, **When** their page script posts `{category:"engagement", action:"click", label:"header-cta"}` to the management endpoint, **Then** the endpoint resolves the visitor's session, inserts exactly one `analyzerCustomEvent` row with that session's `sessionKey`, and returns HTTP 202.
2. **Given** the same payload + a non-null `value` (e.g. `42.5`), **When** the endpoint persists the row, **Then** `value` is stored with full decimal precision (no precision loss beyond `decimal(18,4)`).
3. **Given** a request from a visitor with no active session (no prior pageview in the session window), **When** the endpoint is hit, **Then** the resolver opens a new session synchronously (per slice-003 contract) and the custom event row attaches to that fresh session; the response is still HTTP 202.
4. **Given** an in-process consumer resolves `IAnalyticsEventStateProvider` inside the request scope that handled the custom-event POST, **When** the consumer reads `CurrentRequestCustomEvents`, **Then** the just-persisted `AnalyticsCustomEvent` projection is returned (immutable list of length 1 for a single-call request).
5. **Given** two consecutive `analyzer.send("event",...)` calls from the same page (browser keeps the session warm), **When** the second call lands, **Then** both events attribute to the same session row, `pageviewCount` on the session DOES NOT increment (custom events are NOT pageviews per slice-003 contract), AND `lastActivityUtc` advances to the second event's `receivedUtc` via the new `TouchAsync` repository path (Clarification §1 — custom events keep the session warm).
6. **Given** a page-script call where the user's `analyzer.send(...)` payload is well-formed JSON, **When** Customizer's `PageviewCaptured` has already fired for the current request earlier (pageview capture preceded the script's POST in the same HTTP request lifecycle — atypical but possible for SPA-style flows), **Then** the custom event row's `receiptKey` IS populated with the just-persisted receipt's key; otherwise `receiptKey` remains null.

### User Story 2 — Operator-triggered visitor anonymisation hard-deletes the visitor's custom events (Priority: P2)

When an authorised backoffice user invokes Customizer's "anonymise visitor" operator action, Customizer's `AnonymizeVisitorProfileHandler` runs every registered `IAnonymizationCascadeStep` inside its outer NPoco scope. Analyzer registers an `AnalyzerCustomEventCascadeStep` that **hard-deletes** the visitor's `analyzerCustomEvent` rows. This matches slice-002's receipt cascade-step (hard-delete) rather than slice-003's session cascade-step (soft-anonymise): custom events are per-row engagement signals (e.g. "this visitor clicked the header CTA at 14:32 UTC") with no aggregate-load-bearing role, so removing them on right-to-delete is the simpler + more defensible choice. The slice-002 receipt + slice-003 session cascade steps continue to run as before; the new step joins the cascade-step landscape additively.

**Why this priority**: Constitution Principle IV v1.1.1 mandates that every Analyzer-owned table register a cascade step the moment the table is introduced. Slice 004 introduces `analyzerCustomEvent`, so the cascade step must ship with it; absence is a Constitution Check failure. P2 because operator anonymisation is lower-frequency than the per-page-script-call P1 path.

**Independent Test**: Seed two visitor profiles A and B; drive a small batch of `analyzer.send("event", ...)` calls against each (so both have `analyzerCustomEvent` rows). Invoke Customizer's `AnonymizeVisitorProfileCommand` for visitor A. Re-read `analyzerCustomEvent`: visitor A's rows are gone (zero rows where `visitorProfileKey = A`); visitor B's rows untouched. Also confirm: A's slice-003 sessions are soft-anonymised (preserved per slice-003 contract), A's slice-002 receipts are hard-deleted (per slice-002 contract), Customizer's visitor row is anonymised (per slice-007 contract).

**Acceptance Scenarios**:

1. **Given** `analyzerCustomEvent` rows exist for visitor profiles A and B, **When** an authorised operator anonymises A, **Then** Analyzer's custom-event cascade step deletes every row where `visitorProfileKey = A`, leaves B's rows untouched, and the work completes inside Customizer's outer scope.
2. **Given** the cascade step throws unexpectedly mid-run, **When** the outer handler disposes its scope without calling `Complete()`, **Then** the entire anonymisation rolls back atomically — A's custom-event rows revert (re-appear), A's sessions revert (anonymizedUtc + deviceKey restored), A's receipts revert, Customizer's visitor row is not overwritten.
3. **Given** the cascade step runs against a visitor with zero `analyzerCustomEvent` rows, **When** the step executes, **Then** it returns successfully with zero rows affected and emits no warning.

### User Story 3 — Validation, RBAC, and audit-log on the new management surface (Priority: P3)

The management API endpoint that accepts `analyzer.send(...)` payloads is the first Analyzer-owned management surface to ship (slice 005's content app + slice 010+ reports follow on its conventions). It MUST satisfy Constitution Principle VII end-to-end: anonymous access is rejected; payloads are validated at the boundary (empty `category`/`action` rejected; oversized strings rejected; non-finite numeric `value` rejected; anti-forgery token required on state-changing POSTs); every successful event-recording action emits a structured audit-log entry (visitor `oid` + `upn` per Principle I; action + target; timestamp). Slice 004 codifies the contract; slice 005 reuses it for the content app endpoint.

**Why this priority**: P3 because the failure modes (rejection, audit emission) are operationally important but lower-frequency than the P1 happy path. Principle VII gates require these controls to ship with the management surface; deferring would be a constitution violation at plan time.

**Independent Test**: Send three malformed POSTs to the endpoint: (a) anonymous (no auth cookie), (b) authenticated but with `category = ""`, (c) authenticated with `value = NaN`. Confirm each returns HTTP 401 / 400 with structured error responses (no stack traces leaked); no `analyzerCustomEvent` row is created. Then send one well-formed authenticated POST; confirm the audit-log writer captured an entry containing the actor's UPN, action "custom-event-capture", target = `analyzerCustomEvent`'s new row's `eventKey`.

**Acceptance Scenarios**:

1. **Given** an anonymous request (no authenticated EntraID identity), **When** it POSTs to the management endpoint, **Then** the endpoint returns HTTP 401, emits no audit-log entry, and creates no row.
2. **Given** an authenticated request whose payload has `category = ""` OR `action = ""` OR `category.Length > 64` OR `action.Length > 64` OR `label.Length > 256` OR `value` non-finite (NaN / Infinity), **When** the endpoint validates, **Then** it returns HTTP 400 with a structured error response naming the offending field, emits no audit-log entry, and creates no row.
3. **Given** a POST without a valid anti-forgery token, **When** the endpoint's anti-forgery filter runs, **Then** it returns HTTP 400 (or HTTP 403 depending on framework convention), emits no audit-log entry, and creates no row.
4. **Given** a well-formed authenticated POST with valid anti-forgery, **When** the endpoint completes successfully, **Then** an audit-log entry is emitted carrying the actor's UPN (display) + `oid` (canonical key), action `custom-event-capture`, target = the persisted row's `eventKey`, timestamp = `receivedUtc`.

### Edge Cases

- **`analyzer.send(...)` called before any pageview has been captured this request**: handled by US1 AS3 — the resolver opens a session synchronously. `receiptKey = null` on the row (no parent receipt for an in-request-only custom event).
- **`analyzer.send(...)` called from a request that ALSO captures a pageview**: rare for typical Razor flows (the page render captures the pageview; the script's fetch is a separate request). For SPA-style flows where one request both renders and serves a custom event, `receiptKey` is populated only if the slice-002 pageview handler has already persisted the receipt by the time the custom-event endpoint reads `IAnalyticsEventStateProvider.CurrentRequestReceipt` (typically null per slice-002 contract; populated reliably only on in-request dispatches). Net: `receiptKey` is null in 99% of real flows; populated in the rare in-request co-capture case.
- **Burst of `analyzer.send(...)` calls** (e.g. 50 within 100ms from a misbehaving page script): each lands as a separate `analyzerCustomEvent` row attached to the same session; FR-009 budget (1 SELECT + 1 INSERT per call) holds; no throttling at this layer (operator-side rate-limiting at Umbraco's pipeline would handle abuse).
- **Visitor's session expires between `analyzer.send(...)` calls in a long-lived tab**: the second call's resolver invocation observes the stale session, closes it (lazy-close per slice-003), and opens a new session; the custom event attaches to the new session. Operator sees two session rows for that visitor in a long-tail browsing session.
- **Payload value precision**: `decimal(18,4)` accepts up to 14 digits before the decimal point + 4 after; `value = 1234567890.12345` rounds to `1234567890.1235` per SQL Server's banker's rounding. Documented in the API contract.
- **Operator anonymisation mid-custom-event-burst**: Customizer's orchestrator serialises against the visitor's write queue (slice-007 contract); Analyzer inherits — no race-condition mitigation needed inside the cascade step.
- **Custom event for an anonymised visitor**: if `analyzer.send(...)` arrives for a visitor whose `customizerVisitorProfile.IdentityRef` is `anonymized:...`, the resolver still produces a valid session (per slice-003 — sessions for anonymised visitors are themselves treated as anonymised), and the custom event row persists; downstream queries filter `WHERE anonymizedUtc IS NULL` if they want non-anonymised data.
- **Anti-forgery token missing on a same-origin SPA fetch**: the SPA developer's responsibility to thread the token into the request. Documented in the client-API quickstart; the endpoint rejects with HTTP 400.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Analyzer MUST ship a client API entry point `analyzer.send("event", category, action, label?, value?)` reachable from any authenticated page in the host Umbraco site. The entry point is delivered through Analyzer's existing App_Plugins bundle (slice 001 / slice 005). It MUST POST a JSON payload to the management endpoint described in FR-002, MUST include the standard browser-managed authentication cookie, MUST include the host's anti-forgery token (sourced from the standard Umbraco anti-forgery cookie + header pair), and MUST return `Promise<{ eventKey: string }>` (Clarification §2): the Promise resolves with the server-issued `eventKey` on HTTP 202 and rejects with `{ status: number, message: string }` on any other status (failure logged client-side at `console.warn`; not retried).
- **FR-002**: Analyzer MUST expose a backoffice-authenticated POST endpoint that accepts the JSON payload `{ category: string, action: string, label?: string, value?: number }`. The endpoint is gated by (a) authenticated EntraID session (anonymous rejected per FR-006), (b) anti-forgery token validation (rejected per FR-006), (c) payload validation (per FR-007). On success, it (1) resolves the active session via `IAnalyzerSessionResolver` (synchronously on the request thread; slice-003 contract), (2) persists a single `analyzerCustomEvent` row, (3) updates the request-scoped `AnalyticsEventStateStore` with the in-flight event, (4) emits an audit-log entry (per FR-008), (5) returns HTTP 202 with the new row's `eventKey` in the response body.
- **FR-003**: Analyzer MUST introduce the `analyzerCustomEvent` table with columns: `id` (Guid PK, opaque), `eventKey` (Guid, unique non-clustered, publicly-exposed stable identifier — the value returned in the response body), `sessionKey` (Guid, hard FK to `analyzerSession.sessionKey`), `visitorProfileKey` (Guid, hard FK to `customizerVisitorProfile.key`), `receiptKey` (Guid, NULL allowed — soft FK to `analyzerEventReceipt.id`; indexed for the rare in-request co-capture case), `category` (NVARCHAR(64), NOT NULL), `action` (NVARCHAR(64), NOT NULL), `label` (NVARCHAR(256), NULL), `value` (DECIMAL(18,4), NULL), `receivedUtc` (DateTimeOffset, NOT NULL). Indexes: unique on `eventKey`; non-clustered on `sessionKey` (for session-scoped queries — slice 010+ reports); non-clustered on `visitorProfileKey` (for cascade step's DELETE-by-key); non-clustered on `(category, action)` (for slice-010+ "events by category" aggregation queries).
- **FR-004**: Analyzer MUST ship migration `M0003_AddAnalyzerCustomEventTable` — an additive, idempotent `AsyncMigrationBase` migration that creates `analyzerCustomEvent` per FR-003. FK + index declarations on SQL Server only (SQLite skip per lesson #39). Re-runnable safely via `TableExists` guards. NO back-fill of existing receipts (custom events are forward-only).
- **FR-005**: Analyzer MUST extend the public `IAnalyticsEventStateProvider` (slice-002 + slice-003 pinned contract) with a new read-only member `CurrentRequestCustomEvents` returning `IReadOnlyList<AnalyticsCustomEvent>` (empty list when none captured this request; never null). The new member is purely additive — slice-002's `CurrentRequestReceipt` and slice-003's `CurrentSession` MUST remain unchanged in name, return type, and semantics. The pinning baseline MUST be regenerated to capture the new member + the new `AnalyticsCustomEvent` record.
- **FR-006**: Analyzer MUST reject anonymous requests to the management endpoint with HTTP 401 (no row created; no audit entry). Anti-forgery token validation MUST run before payload validation; missing/invalid tokens MUST return HTTP 400 (or 403 per Umbraco/ASP.NET Core convention). The endpoint MUST NOT bypass Umbraco's standard backoffice authentication primitives (Principle VII).
- **FR-007**: Payload validation MUST reject (HTTP 400, structured error naming the offending field): empty `category`; empty `action`; `category.Length > 64`; `action.Length > 64`; `label.Length > 256` (label is nullable but if present non-empty must fit); `value` non-finite (NaN, ±Infinity); `value` precision exceeding `decimal(18,4)` after parsing. Whitespace-only `category`/`action` are treated as empty. Validation runs at the request boundary (controller filter or model-state validation per ASP.NET conventions); the domain layer MUST NOT trust upstream validation.
- **FR-008**: Every successful event-recording action MUST emit a structured audit-log entry capturing: actor (`UPN` for display + `oid` for canonical key per Principle I), action name `"custom-event-capture"`, target = the persisted row's `eventKey`, timestamp = `receivedUtc`, optional `category` + `action` (for operator-side correlation). Failed validations DO NOT emit audit entries (they're noise; structured logs at warn-level capture them for operator visibility).
- **FR-009**: Analyzer MUST register a third `IAnonymizationCascadeStep` (`AnalyzerCustomEventCascadeStep`) alongside slice-002's receipt cascade-step + slice-003's session cascade-step. The new step MUST execute inside Customizer's outer NPoco scope BEFORE the visitor-row overwrite. The step MUST **hard-delete** rows for the supplied `visitorProfileKey` (matches slice-002 receipt cascade pattern; Principle IV v1.1.1 authorises per-table choice). Step ordering relative to slice-002 + slice-003 cascade steps is irrelevant because all three operate on disjoint tables and participate in the same outer transaction.
- **FR-010**: The per-call cost on the request thread MUST be bounded to ≤ 1 indexed SELECT (resolver cache miss; typically 0 SELECTs on warm cache) + ≤ 1 indexed UPDATE via `IAnalyzerSessionRepository.TouchAsync` (cache-hit path advances `lastActivityUtc` only; no `pageviewCount` change per Clarification §1) + ≤ 1 indexed INSERT (`analyzerCustomEvent` row) + ≤ 1 audit-log emit (in-process; backed by `ILogger`). Total worst-case ≤ 3 SQL statements per call.
- **FR-011**: Analyzer MUST publish a new public record type `Analyzer.Analytics.AnalyticsCustomEvent` exposing `(EventKey: Guid, SessionKey: Guid, VisitorProfileKey: Guid, ReceiptKey: Guid?, Category: string, Action: string, Label: string?, Value: decimal?, ReceivedUtc: DateTimeOffset)`. The record MUST live in the `Analyzer.Analytics` namespace alongside `AnalyticsEventReceipt`, `AnalyticsSession`, and `IAnalyticsEventStateProvider` so the pinning baseline captures it directly.
- **FR-012**: Cited reference-doc parity items: `FR-EVT-04` (custom-event capture — operator-defined name/category/action/label/value), `FR-EVT-05` (custom-event dimensions — visitor + session + content node correlation; this slice ships the visitor + session correlation; content-node correlation lands when the receipt's content-node column is surfaced in a later slice), `FR-COL-01` (server-side capture continues — inherited from slice 002 + slice 003 with no regression), `FR-SRC-01` (operator-defined event sources — the `category` + `action` fields are the operator-defined source). None of the dropped prefixes (`FR-DEP-*`, `FR-DIM-04`, `FR-DIM-03`, §3.3 bot detection, §6.2 public-website features) are cited.
- **FR-013**: All slice-004 public types — `IAnalyticsEventStateProvider` (modified — `CurrentRequestCustomEvents` added), `AnalyticsCustomEvent` (new), plus the slice-001 + slice-002 + slice-003 types already pinned (`AnalyticsEventReceipt`, `AnalyticsSession`, `IVisitorIdentifier`, `VisitorIdentity`) — MUST be captured by `PublicSurfacePinningTests`. The slice-004 baseline regeneration MUST be documented in the existing spec.md Assumptions entry titled "Public-surface pinning regeneration" with the specific additive diff lines, consistent with slice 002 + slice 003 precedent.

### Key Entities

- **Custom event (Analyzer-owned, new this slice)**: A single operator-defined engagement signal recorded by a client-side `analyzer.send(...)` call. Keyed by an opaque `id` PK + a publicly-exposed stable `eventKey`. Hard FKs to `analyzerSession.sessionKey` AND `customizerVisitorProfile.key`. Soft FK to `analyzerEventReceipt.id` (nullable; populated only on the rare in-request co-capture). Carries the operator-defined dimensions (`category`, `action`, `label`, `value`) + the capture timestamp.
- **Custom-event state surface (Analyzer-owned, modified this slice)**: `IAnalyticsEventStateProvider` gains `CurrentRequestCustomEvents`. The slice-002 `CurrentRequestReceipt` + slice-003 `CurrentSession` members are unchanged. The backing `AnalyticsEventStateStore` gains a parallel `_currentCustomEvents` list field + `AppendCustomEvent(AnalyticsCustomEvent)` mutator. Within a single request scope, the list grows as the page script makes multiple `analyzer.send(...)` calls (rare but supported).
- **Session (Analyzer-owned, read-only at this slice)**: custom events FK to `analyzerSession.sessionKey`. The session is opened-or-extended by the slice-003 resolver synchronously when the custom-event endpoint runs.
- **Visitor profile (read-only at this slice, owned by Customizer)**: custom events FK to `customizerVisitorProfile.key`. Identity claims (`oid` + `upn`) are surfaced to the audit-log writer via the slice-001 `IVisitorIdentifier`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For an authenticated EntraID employee firing `analyzer.send("event", "engagement", "click", "header-cta")` N consecutive times, exactly **N** rows appear in `analyzerCustomEvent`, all sharing the same `sessionKey`, ordered by `receivedUtc` ascending. (Test cases: N=1, N=3, N=10.)
- **SC-002**: For an authenticated EntraID employee firing a custom event AFTER their existing session has expired (longer than the configured `InactivityTimeoutMinutes` since the last activity), exactly **two** `analyzerSession` rows are observed for that visitor — the old one closed by the resolver's lazy-close branch, the new one carrying the custom event row's `sessionKey`.
- **SC-003**: Under sustained load matching slice-002's published envelope (1000 events/s for 60 seconds), the 95th-percentile request-thread latency on the management endpoint is **≤ 5 ms** (single SQL round-trip for the resolver-cache-hit path + single INSERT for the row). Cache-miss path p95 is **≤ 12 ms**.
- **SC-004**: Operator-triggered visitor anonymisation hard-deletes **all** of the visitor's `analyzerCustomEvent` rows in **≤ 200 ms for a visitor carrying ≤ 1 000 custom events** (matching slice-002's SC-003 budget shape). Post-anonymisation: zero rows where `visitorProfileKey = <anonymised key>`.
- **SC-005**: Anonymous POSTs to the management endpoint return HTTP 401 in **100% of cases** (zero false-accepts); zero `analyzerCustomEvent` rows are created from anonymous requests.
- **SC-006**: Validation failures (empty fields, oversized strings, non-finite numerics) return HTTP 400 with a structured error response in **100% of cases**; zero `analyzerCustomEvent` rows are created from invalid payloads. Validation runs in **≤ 1 ms** on the request thread (no I/O).
- **SC-007**: Every successful capture emits exactly **one** audit-log entry containing the actor's UPN + oid + `eventKey` + timestamp; zero successful captures occur without a matching audit entry.
- **SC-008**: The public-surface pinning baseline regenerates without diff against the checked-in baseline. The baseline includes `IAnalyticsEventStateProvider` with all three members (`CurrentRequestReceipt`, `CurrentSession`, `CurrentRequestCustomEvents`), `AnalyticsCustomEvent` as a new public record, and slice-002 + slice-003 types unchanged.
- **SC-009**: Slice 004 ships with passing unit and integration tests covering all US1 acceptance scenarios (1–6), all US2 acceptance scenarios (1–3), and all US3 acceptance scenarios (1–4). Integration tests run against a real SQL Server container (matching slice 002 + slice 003 pattern).

## Assumptions

- **Management endpoint route prefix**: The slice-005-anticipated Analyzer backoffice route prefix is not yet pinned (per CLAUDE.md "concrete management-API route prefix is deliberately not declared yet"). Slice 004 ships the endpoint behind a slice-004-local placeholder prefix (to be pinned in plan.md per the constraint that the prefix is NOT `/umbraco/engage/...`). Slice 005 may rename the prefix; the rename is a route change only (no semantic impact on the endpoint contract).
- **Client API delivery mechanism**: The `analyzer.send(...)` entry point is delivered via the existing App_Plugins bundle. Slice 001 ships an empty-token bundle; slice 005 wires the backoffice content app's TypeScript work. For slice 004, the entry point is added to the existing Analyzer bundle as a globally-exposed function on `window.analyzer.send`. The bundle stays a single `analyzer.js` artefact; no separate front-end-facing CDN endpoint is introduced.
- **Cascade-step semantic is hard-delete** (Principle IV v1.1.1 per-table choice): custom events are per-row engagement signals; deleting on anonymisation matches the slice-002 receipt cascade. The session cascade's soft-anonymise pattern doesn't apply because no aggregate query depends on the deleted rows surviving (per-content-node aggregates land in slice 005 and project from the receipt + session tables, NOT custom events).
- **Custom-event writes are synchronous to the request thread** (NOT queued via a bounded channel like the slice-002 receipt path): the page script's POST is a stand-alone HTTP request; the response is the contract that the row is recorded. Asynchronous enqueue + later flush would force HTTP 202 to mean "accepted but maybe dropped" which violates the operator's expectation. The per-call cost (≤ 1 SELECT + ≤ 1 UPDATE + 1 INSERT) is bounded; the resolver's LRU cache amortises the SELECT across the warm working set.
- **No throttling/rate-limiting at this layer**: a misbehaving page script firing 100 events/s gets 100 rows per second. Umbraco's pipeline + ASP.NET Core's standard rate-limiter (when configured by the host) is the boundary control; Analyzer does not introduce its own. Documented in the API contract.
- **Audit-log substrate**: structured logging via `ILogger<AnalyzerCustomEventController>` (or equivalent) with named properties `Actor`, `ActorOid`, `Action`, `Target`. Slice 004 does not introduce a dedicated `analyzerAuditLog` table; that lands in a later slice if compliance posture requires a queryable trail. For now, log-shipping (e.g. Serilog → Elastic, App Insights) is the operator-side audit substrate, consistent with Customizer's slice-002 webhook audit pattern.
- **Anti-forgery integration**: the existing Umbraco backoffice anti-forgery infrastructure (cookie + header pair) is the source of truth. Slice 004's endpoint is registered under the standard Umbraco backoffice management route convention, so the framework's default anti-forgery filter applies automatically. No new anti-forgery code path is introduced.
- **`receiptKey` correlation is best-effort**: read `IAnalyticsEventStateProvider.CurrentRequestReceipt` at endpoint entry; if non-null, attach it to the custom-event row; else null. The receipt's in-request state is reliably populated only for in-request consumers (per slice-002 + slice-003 contracts), so non-null `receiptKey` is the rare-but-valid case. Operator-side queries treat `receiptKey IS NULL` as the common case.
- **Public-surface pinning regeneration**: slice 004 changes `IAnalyticsEventStateProvider` additively (new `CurrentRequestCustomEvents` member) AND adds a new `AnalyticsCustomEvent` record. The baseline file at `src/Analyzer.Tests/PublicSurface/Baselines/Analyzer-public-surface.txt` is regenerated as part of this slice; the additive change is documented in the existing Assumptions entry titled "Public-surface pinning regeneration" (slice-002 + slice-003 precedent). The change is MINOR-additive (no breaking member rename or removal), consistent with Constitution Principle X.
- **No new cross-product Customizer prereq**: verified — slice 003 + slice 002 already exposed `IAnalyzerSessionResolver`, `IAnalyticsEventStateProvider`, `customizerVisitorProfile`, `IAnonymizationCascadeStep`. Slice 004 depends only on the surfaces shipped through customizer `5273c38` (`Pageview.UserAgent`) + customizer `05e989c` (`PageviewCaptured`) + customizer slice 007 (`IAnonymizationCascadeStep`). No Customizer-side change is required.
- **No backoffice UI yet**: slice 004 ships zero `@umbraco-cms/backoffice` components. The TypeScript work in `src/Analyzer/Client/` for slice 004 is limited to extending the App_Plugins bundle's exposed `window.analyzer` object with the `send(...)` method + a thin fetch wrapper for the management endpoint. The operator-facing "configure custom event types" surface + "view custom events per visitor" surface land in slice 005 / slice 010+.
- **No reports**: slice 004 ships no operator-facing aggregation. Custom events become queryable via SQL but no backoffice surface aggregates them. The `(category, action)` index supports the eventual "events by category" report (slice 010+).
- **Custom events do not increment session pageviewCount, but DO advance `lastActivityUtc`** (Clarification §1): per slice-003 contract, `pageviewCount` is the count of pageview events attached to the session — custom events do not bump it.
