# Feature Specification: Pageview Subscription + Analytics-Event State Provider

**Feature Branch**: `002-pageview-subscription`

**Created**: 2026-05-18

**Status**: Draft

**Input**: User description: "pageview subscription — Analyzer subscribes to Customizer's `PageviewCaptured` INotification and persists an analyzer-side analytics-event row keyed by visitor identity (oid-first, upn-fallback) with foreign-key references back to the Customizer pageview row. Introduces Analyzer's `IAnalyticsEventStateProvider` request-state contract."

## Clarifications

### Session 2026-05-18

- Q: What is the retention / pruning scope for slice 002? → A: Defer. Slice 002 ships no retention policy or pruning job. The event-receipt table carries an `IDX_ReceivedUtc` index so a later slice's pruning job can scan by date efficiently, but no policy ships now. The cross-cutting retention design consolidates in a later slice (anticipated: slice 013, "Anonymisation cascade-step registrations," per inter-product contract §4) when sessions, custom events, video, forms, scroll, and search tables all exist and can be pruned by a single policy. Constitution Principle VII's "configurable retention" obligation is satisfied at the principle layer.
- Q: What is the receipt-write durability promise under back-pressure and crash? → A: At-most-once delivery via an in-memory bounded queue, mirroring Customizer's `FR-025` envelope. Under back-pressure (queue full): drop with a warning log. On crash: in-flight receipts are lost. No persistent queue, no synchronous-insert path. The receipt is an audit row whose value lives in aggregate counts over time, not in any single row, so the loss profile is acceptable; matching Customizer's promise keeps the two products' loss-tolerance stories aligned and preserves the 1000 pv/s sustained throughput target without fsync overhead.
- Q: What is the public-surface pinning baseline scope for slice 002? → A: Pin Analyzer's named extension contracts only — `Analyzer.Analytics.IAnalyticsEventStateProvider` (new this slice), plus `Analyzer.Identity.IVisitorIdentifier` and `Analyzer.Identity.BaseVisitorIdentifier` (from slice 001, unpinned until now per slice 001 Clarification Q2). Do NOT pin the cascade-step implementation class (internal-by-convention), the event-receipt entity (internal storage shape), or `AnalyzerComposer` (Umbraco-discovered, not consumer-imported). Matches Constitution Principle X's "extension surface" framing — pinning covers what third-party developers can implement and register against. Re-evaluate scope when `IEventDimensionExtractor` lands (slice 014, per inter-product contract §4).

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Analyzer records an event receipt for every captured pageview (Priority: P1)

When the host renders a page for an authenticated EntraID employee, Customizer's pageview-capture middleware publishes a `PageviewCaptured` notification carrying the captured `Pageview` snapshot. Analyzer subscribes to that notification, derives the visitor identity from the carried `VisitorProfileKey` (the oid-first canonical key already pinned by Customizer's identity layer), and records an Analyzer-owned event-receipt row referencing the parent pageview. This is the first slice where Analyzer writes data; the receipt row is the substrate every later Analyzer event table (sessions, custom events, video, scroll, forms, search) extends.

**Why this priority**: Every later Analyzer slice depends on a working pageview-subscription channel. Without it, Analyzer has no entry point into the request stream and no Analyzer-owned data exists to demonstrate Principle IV's FK invariant or Principle VIII's hot-path budget. This story delivers the load-bearing capture path and is the minimum viable definition of "Analyzer records data."

**Independent Test**: An authenticated EntraID employee navigates to a page on a host running Customizer + Analyzer. A read of Analyzer's event-receipt table after the request returns exactly one row whose visitor-profile reference matches the EntraID `oid`-resolved visitor profile and whose pageview reference matches the `customizerPageview` row Customizer wrote for that same request.

**Acceptance Scenarios**:

1. **Given** an authenticated EntraID request with both `oid` and `upn` claims, **When** the host renders the page and Customizer's middleware publishes `PageviewCaptured`, **Then** Analyzer's subscriber executes asynchronously off the request thread, derives the visitor identity through the existing Analyzer identity seam, and writes exactly one event-receipt row referencing the captured pageview key and the visitor's profile key.
2. **Given** Customizer drops the pageview persistence under back-pressure (per Customizer's `FR-025` write-queue rules — the notification still fires but the parent `customizerPageview` row will not exist), **When** Analyzer's subscriber processes the notification, **Then** the subscriber records its receipt against the visitor profile key only, without producing a foreign-key violation against the absent pageview row, and emits a debug-level log indicating the back-pressure-drop case.
3. **Given** the same `PageviewCaptured` notification is dispatched more than once for the same `Pageview.Key` (per Customizer's fire-and-forget contract, duplicates are theoretically possible under unusual scheduling), **When** Analyzer's subscriber executes the second dispatch, **Then** no duplicate receipt row is created and no exception propagates to the caller.
4. **Given** Analyzer's subscriber encounters a transient database error while writing the receipt, **When** the handler returns, **Then** the error is logged at warning level, the request thread is not affected (the handler runs on Customizer's fire-and-forget dispatch), and Customizer's middleware proceeds normally.

### User Story 2 — Operator-triggered visitor anonymisation cascades into Analyzer's event-receipt rows (Priority: P2)

When an authorised backoffice user invokes Customizer's "anonymise visitor" operator action (for example in response to a CCPA right-to-delete request the deploying organisation is fulfilling), Customizer's anonymisation orchestrator runs every registered `IAnonymizationCascadeStep` in turn. Analyzer registers a cascade step for its event-receipt table so the matching rows are deterministically re-keyed to the anonymised visitor profile key — preserving aggregate counts while severing the link to the original EntraID identity. Without this cascade step, Analyzer rows would carry stale FKs after Customizer's anonymisation runs, breaking the FK invariant and the erasure-compliance posture.

**Why this priority**: Constitution Principle IV mandates that every Analyzer-owned table register a cascade step the moment the table is introduced. Slice 002 introduces the first such table, so the cascade step must ship with it; absence is a Constitution Check failure. This story is P2 (not P1) only because operator anonymisation is a lower-frequency event than pageview capture — both ship in this slice.

**Independent Test**: Seed two visitor profiles, capture a small set of pageviews against each, verify Analyzer's event-receipt rows reference the original visitor profile keys, then invoke Customizer's `AnonymizeVisitorProfileCommand` for one of the profiles. Re-read Analyzer's event-receipt rows and assert: the anonymised visitor's rows now reference the anonymised key (or are deleted, per the cascade-step's chosen semantic); the untouched visitor's rows are unchanged; and aggregate counts (rows-per-visitor) are preserved exactly.

**Acceptance Scenarios**:

1. **Given** Analyzer event-receipt rows exist for two visitor profiles A and B, **When** an authorised operator invokes Customizer's anonymise action for visitor A, **Then** Analyzer's cascade step re-keys every event-receipt row formerly referencing A's profile key, leaves B's rows untouched, and the cascade step's outcome is reported back through Customizer's anonymisation result envelope.
2. **Given** Analyzer's cascade step throws unexpectedly, **When** the orchestrator collects step outcomes, **Then** the failure is reported in the result envelope with a step identifier, the orchestrator does not silently swallow the error, and subsequent steps still run (per Customizer's existing slice-007 cascade orchestration semantics).
3. **Given** the cascade step runs against a visitor with zero Analyzer event-receipt rows, **When** the step executes, **Then** it returns a success outcome with `RowsAffected = 0` and emits no warning.

### User Story 3 — Other Analyzer services read the current request's analytics-event state through a stable contract (Priority: P3)

A developer working on a later Analyzer slice (sessions, custom events, content app, reports) needs to know what Analyzer-side data has been captured for the in-flight request — for example "did Analyzer's subscriber record a receipt for this pageview yet?" or "what custom events has the client fired this request?". This slice introduces the `IAnalyticsEventStateProvider` interface — a request-scoped, read-only contract that exposes Analyzer's view of the current request's captured analytics-event state. The contract is intentionally separate from Customizer's pinned `IAnalyticsStateProvider` (which exposes visit-history aggregates and personalization-profile reads); both interfaces can be injected into the same consumer without collision.

**Why this priority**: P3 because no shipped Analyzer surface in slice 002 consumes the provider; it is introduced as an extension surface that later slices (3 onwards) plug into. Shipping the contract now establishes the public-surface pinning that Constitution Principle X requires before later slices announce stable behaviour, and locks the name + DI lifetime before any consumer commits to it.

**Independent Test**: A test resolves `IAnalyticsEventStateProvider` from DI within an authenticated request scope, asserts the resolved instance is scoped (the same instance is returned for two resolutions inside the same scope but a different instance is returned in a separate scope), and reads the current request's event-receipt reference — returning the in-flight receipt when the subscriber has run and `null`/empty when it has not.

**Acceptance Scenarios**:

1. **Given** an authenticated request reaches the rendering pipeline, **When** a consumer resolves `IAnalyticsEventStateProvider` from `IServiceProvider` within that request scope, **Then** a non-null scoped instance is returned and its current-pageview member exposes the event-receipt the subscriber recorded for this request (or `null` if the subscriber has not yet executed, given the fire-and-forget dispatch).
2. **Given** two separate authenticated requests are in flight concurrently, **When** each request reads the provider, **Then** each sees only its own request's state — no cross-request leakage (Constitution Principle VIII: no per-process in-memory state that cannot be scoped).
3. **Given** Analyzer's public surface is pinned, **When** the public-surface pinning test suite is executed, **Then** the namespace, type name, and member shape of Analyzer's named extension contracts — `IAnalyticsEventStateProvider`, `IVisitorIdentifier`, and `BaseVisitorIdentifier` (per Clarifications Q3) — are captured in the pinning snapshot so future slices cannot silently break any of them (Principle X).

### Edge Cases

- **Notification fires but `VisitorProfileKey` is `Guid.Empty`**: the `Pageview` record carries a non-nullable `VisitorProfileKey: Guid` by contract; Analyzer can rely on it being present. If a future Customizer change or a misconfigured external-login provider ever surfaces `Guid.Empty`, Analyzer's handler treats that as a configuration-error fallback case (per Constitution Principle I), logs a warning, and skips the write — there is no anonymous-fallback Analyzer row.
- **Customizer absent at boot**: composition already fails fast under slice 001's `FR-002`. Slice 002 inherits that contract — there is no "Analyzer runs without Customizer" runtime path to test.
- **Analyzer database schema not migrated**: at first boot, Umbraco's standard migration pipeline applies Analyzer's EF migration before the subscriber handles its first notification. If a host operator skips migrations, the subscriber logs a single startup-time error referencing the missing table and disables itself for the lifetime of the process rather than retry-flooding the log.
- **High-volume pageview burst (≥ 5000 pv/s peak per Customizer's `NFR` envelope)**: Analyzer's subscriber MUST not regress Customizer's throughput. The handler is async, performs no synchronous I/O on the request thread, and writes its receipt via a bounded in-memory queue + dispatcher pattern (mirroring Customizer's slice-003 write-queue discipline) — not a per-notification synchronous DB insert.
- **Bounded write-queue is full (back-pressure)** (Clarifications Q2): Analyzer's dispatcher MUST drop the notification with a warning log capturing the offending `Pageview.Key` and the queue's current depth. At-most-once delivery semantics — no retry, no persistent fallback. Mirrors Customizer's `FR-025` envelope. Drop accounting is not a metric in slice 002 (deferred to a later observability slice).
- **Host crash mid-flush** (Clarifications Q2): in-flight receipts in the bounded queue are lost. No durability promise per individual receipt; the receipt table's *aggregate counts over time* are the load-bearing artefact, not any single row's existence.
- **Customizer's anonymisation runs concurrently with a pageview capture for the same visitor**: ordering is determined by Customizer's orchestrator (the anonymisation command serialises against the visitor's write queue per Customizer's slice-007 contract). Analyzer's cascade step inherits that serialisation; no Analyzer-side locking is required.
- **Subscriber handler throws**: the handler MUST catch its own exceptions and log them at warning level; the exception MUST NOT propagate through Umbraco's `IEventAggregator` to Customizer's middleware (per Customizer's `PageviewCaptured` fire-and-forget contract).
- **Duplicate notification for the same `Pageview.Key`**: handler is idempotent — the database unique constraint on `PageviewKey` (within the receipt table) absorbs duplicates and the handler tolerates the resulting unique-violation as a no-op success.
- **Reading the state provider before the subscriber has executed**: legal — the provider returns `null` for the current-pageview receipt. Consumers must tolerate the "subscriber has not yet handled this request" state, because Customizer's fire-and-forget dispatch does not guarantee subscriber completion before the request thread continues.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Analyzer MUST register an `INotificationAsyncHandler<PageviewCaptured>` against Customizer's `PageviewCaptured` notification (Customizer's slice-011 contract at commit `05e989c`), wired through the Analyzer composer (slice 001 `AnalyzerComposer`) ordered after `VisitorAnalyticsComposer` (per slice 001 `ComposeAfter` convention).
- **FR-002**: For every `PageviewCaptured` dispatched, Analyzer MUST persist exactly one event-receipt row referencing the carried `Pageview.Key` (as an unconstrained reference column — soft FK, indexed) and the carried `Pageview.VisitorProfileKey` (as a foreign key — hard FK — to `customizerVisitorProfile.Key`). The asymmetric FK strategy is required because `PageviewCaptured` is published before — and independently of — `customizerPageview` row persistence (per `PageviewCaptured` remarks: notifications fire even when the write queue drops the row under back-pressure). The visitor profile row is guaranteed to exist by the time the notification fires; the pageview row is not.
- **FR-003**: The event-receipt row MUST carry, at minimum: a primary key (Guid), the soft pageview reference (`PageviewKey: Guid`), the hard visitor-profile FK (`VisitorProfileKey: Guid`), the capture timestamp (`ReceivedUtc: DateTimeOffset`). The receipt table MUST carry an index on `ReceivedUtc` (`IDX_ReceivedUtc`) so a later slice's date-range pruning job can scan efficiently without a table scan — slice 002 itself ships no pruning, per Clarifications. Later slices may add columns additively per Constitution Principle X; slice 002 ships only this minimal set.
- **FR-004**: The receipt write path MUST be idempotent on `PageviewKey` — duplicate dispatches MUST NOT produce duplicate rows. Idempotency is enforced by a unique index on `PageviewKey`; the handler MUST tolerate the resulting unique-violation as a successful no-op (per Customizer's `PageviewCaptured` "handlers SHOULD be idempotent" subscriber contract).
- **FR-005**: The subscriber handler MUST NOT propagate exceptions to Customizer's notification dispatch. Any handler exception MUST be caught, logged at warning level with the offending `Pageview.Key`, and swallowed so Customizer's fire-and-forget contract is honoured and a single Analyzer fault does not stop other subscribers (per the `PageviewCaptured` remarks block).
- **FR-006**: Analyzer MUST register an `IAnonymizationCascadeStep` (the Customizer extension contract from `Customizer.Features.Visitors.Application.Contracts.Anonymization`) that re-keys event-receipt rows when Customizer's `AnonymizeVisitorProfileCommand` executes against a visitor profile. The step MUST report its `RowsAffected` count back through the orchestrator's result envelope and MUST be discoverable by Customizer's orchestrator at startup. Absence of this cascade step is a Constitution Principle IV violation (no new Analyzer table without cascade-step registration). Per `NFR-SEC-04`.
- **FR-007**: Analyzer MUST publish a new public type, `Analyzer.Analytics.IAnalyticsEventStateProvider`, exposing read-only access to the current request's Analyzer-side captured state. The interface MUST be registered as **scoped** (per-request) in the DI container and MUST be invocable inside any Analyzer or Customizer request scope without ambiguity against Customizer's pinned `IAnalyticsStateProvider` (per inter-product contract D3). Slice 002's surface exposes at minimum the current request's event-receipt reference; later slices add members additively per Principle X.
- **FR-008**: The migration that creates the event-receipt table MUST run under Umbraco's standard migration pipeline (via an `IMigrationPlan` or equivalent), MUST be re-runnable safely, and MUST run before the subscriber handles its first notification on a fresh host install. The migration's hashed identifier MUST be added to Analyzer's migration history table so a re-deploy never re-applies it.
- **FR-009**: Slice 002 MUST introduce a `PublicSurfacePinningTests`-style pinning test that captures the namespace, type name, and member shape of Analyzer's named extension contracts (per Constitution Principle X and slice 001 Clarification Q2 — slice 002 is where the "richer surface worth pinning" arrives). The pinned set covers `Analyzer.Analytics.IAnalyticsEventStateProvider` (new this slice) plus `Analyzer.Identity.IVisitorIdentifier` and `Analyzer.Identity.BaseVisitorIdentifier` (from slice 001, retroactively pinned now). The pinning explicitly excludes the cascade-step implementation class, the event-receipt entity, and `AnalyzerComposer` — per Clarifications Q3. The pinning baseline MUST be checked in alongside the slice; the scope is re-evaluated when `IEventDimensionExtractor` lands (slice 014).
- **FR-010**: All Analyzer subscriber, cascade-step, and state-provider operations MUST run on the request hot path with negligible latency relative to Customizer's baseline. The subscriber MUST NOT regress Customizer's published throughput envelope (1000 pv/s sustained, 5000 pv/s peak) — per Constitution Principle VIII and inter-product contract D2. The subscriber MUST enqueue the receipt into a bounded in-memory dispatcher queue and return; it MUST NOT perform a synchronous DB insert on the notification dispatch path. When the bounded queue is full, the dispatcher MUST drop the receipt with a warning log capturing `Pageview.Key` (at-most-once delivery, mirroring Customizer's `FR-025`).
- **FR-011**: Cross-product asynchronous work emitted by Analyzer (none in slice 002 itself, but the discipline is established here) MUST go through Customizer's slice-002 outbox dispatcher rather than a separate Analyzer-side transport — per Constitution Principle VIII and inter-product contract D7. Slice 002 introduces no new outbox event types; later slices that do MUST register them via Customizer's event registry per the same contract.
- **FR-012**: Cited reference-doc parity items: `FR-COL-01` (server-side pageview capture is satisfied end-to-end through Customizer's middleware + Analyzer's subscriber), `FR-COL-04` (data stays inside the host Umbraco database — Analyzer writes only to the same `IUmbracoDatabase`), `FR-IDP-04` (Analyzer's custom extension integrates the authenticated identity into the analytics pipeline at request time, via Customizer's already-resolved `VisitorProfileKey`), `FR-ID-05` (no anonymous fallback — handler skips writes when `VisitorProfileKey` is `Guid.Empty`, the only configuration-error fallback shape this slice models).

### Key Entities *(include if feature involves data)*

- **Analytics-event receipt (Analyzer-owned, new this slice)**: A per-pageview audit row recording that Analyzer's subscriber observed and processed a captured pageview. Keyed by an opaque primary key (Guid); references the carried `Pageview.Key` as an indexed unconstrained column (soft FK, because Customizer may drop the parent row under back-pressure) and `customizerVisitorProfile.Key` as a hard foreign key (the visitor row is guaranteed to exist by the time the notification fires). Stores the capture timestamp. Future slices extend additively — slice 003 (sessions) may add a `SessionKey`, slice 004 (custom events) introduces a separate but related table, etc. The receipt is the substrate that lets the per-content-node Analytics content app and reports (D9) attribute aggregate counts back to a visitor without re-deriving identity each query.
- **Visitor profile (read-only at this slice, owned by Customizer)**: Canonical via slice 001 `IVisitorIdentifier`. Analyzer's receipt FK's to `customizerVisitorProfile.Key` directly; the slice never inspects `IdentityRef` (the log-only string) and never duplicates the row.
- **Pageview (read-only at this slice, owned by Customizer)**: `customizerPageview` is the canonical pageview row; Analyzer references its `Key` as a soft pointer only (no FK constraint). The `Pageview` record's other fields (`ContentKey`, `RequestUtc`, `Segments`, UTM trio) are not stored on the receipt — they remain available via join to `customizerPageview` when needed (and absent when Customizer dropped the row).

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For every authenticated EntraID pageview rendered against a host running Customizer + Analyzer with this slice installed, an Analyzer event-receipt row exists within **1 second of the request thread completing**, attributable to the correct visitor profile (100% of the test cases covering the four documented authenticated identity shapes — `oid+upn`, `upn-only` with warning, `oid-only`, anonymized — from slice 001's `IVisitorIdentifier` contract).
- **SC-002**: Under sustained load matching Customizer's slice-003 published envelope (**1000 pv/s for 60 seconds**), Analyzer's subscriber processes ≥ 99% of dispatched notifications to receipt persistence (the ≤ 1% loss budget accommodates the at-most-once back-pressure drop semantic per Clarifications Q2), the 95th-percentile request latency on the rendering thread does **not** regress by more than **2 ms** relative to a Customizer-only baseline, and no notification dispatch is observed to block the request thread.
- **SC-003**: Operator-triggered visitor anonymisation completes the Analyzer cascade step in **≤ 200 ms for any single visitor profile carrying ≤ 10 000 receipt rows**, and the post-anonymisation row count attributable to the original visitor profile key is **0** (rows are either re-keyed to the anonymised key or deleted, per the chosen semantic — both leave zero rows on the original key).
- **SC-004**: Duplicate `PageviewCaptured` dispatches for the same `Pageview.Key` produce **exactly one** receipt row in 100% of test cases (idempotency is observable, not merely defended).
- **SC-005**: The public-surface pinning test passes with a checked-in baseline covering `IAnalyticsEventStateProvider`, `IVisitorIdentifier`, and `BaseVisitorIdentifier` (per Clarifications Q3); any change to those types' namespace, type name, or member shape fails the pinning test (per Constitution Principle X).
- **SC-006**: Slice 002 ships with passing unit and integration tests covering all subscriber acceptance scenarios (US1 1–4), all cascade-step acceptance scenarios (US2 1–3), and all state-provider acceptance scenarios (US3 1–3). Integration tests run against a real SQL Server container (matching the Aspire-orchestrated development topology) — not a SQLite seam — because the cascade step's behaviour under Customizer's orchestrator is the load-bearing assertion and depends on real transaction semantics.
- **SC-007**: An author opening slice 003 (sessions) has zero remaining setup work to subscribe to pageview captures, resolve visitor identity, or write an Analyzer-owned table — slice 003's work begins at "session aggregation rule + close-job" with no scaffolding churn.

## Assumptions

- **Customizer slice 011 is already on `main`** (commit `05e989c`): `PageviewCaptured` is published on every successful capture, fire-and-forget via `Task.Run` dispatch. Slice 002 does not gate on a paired Customizer commit per inter-product contract §6 item 1 — that prerequisite landed last cycle.
- **`PageviewCaptured` is fire-and-forget**: the notification fires regardless of whether the parent `customizerPageview` row persists. Analyzer's receipt-write strategy (soft FK on `PageviewKey`, hard FK on `VisitorProfileKey`) is the direct consequence — back-pressure-dropped pageviews still produce receipts, joinable later only if the pageview row exists.
- **Visitor profile is guaranteed to exist** by the time `PageviewCaptured` fires: Customizer's middleware resolves the profile during request processing before notification dispatch, and the profile's persistence is synchronous to the request thread (unlike the pageview's, which goes through a write queue). This is the foundation of the hard-FK choice on `VisitorProfileKey`.
- **Idempotency key is `Pageview.Key`**: a single pageview produces exactly one Analyzer receipt regardless of dispatch count. Customizer's `Pageview.Key` is the public identifier carrier (`FR-006` in Customizer's slice-003) and is generated once per capture, so it is a safe natural idempotency key.
- **Cascade-step semantic is re-key, not delete**: aggregate report counts (per-content-node pageview totals, visitor-count aggregates) MUST survive an anonymisation event; deleting receipts would distort historical aggregates retroactively. Re-keying to the anonymised profile key preserves the row's contribution to aggregates while severing the EntraID link. This matches Customizer's slice-007 cascade semantics for `customizerGoalReached` (re-key, not delete).
- **The Aspire-orchestrated SQL Server container is the integration-test substrate**: slice 001's lesson #19 establishes the persistent-volume topology; slice 002's integration tests reuse it via Testcontainers or the existing AppHost. SQLite is not viable for this slice because the cascade-step's transaction semantics under Customizer's orchestrator are not faithfully replicated on SQLite.
- **Public-surface pinning baseline is created as part of this slice** (per slice 001 Clarification Q2 deferral, scope refined in Clarifications Q3): slice 002 is the first slice introducing a surface worth pinning. The pinning approach mirrors Customizer's `PublicSurfacePinningTests` — snapshot type and member shape, fail on diff, regenerate on intentional change with a justified `Sync Impact`-style note in the spec. Scope covers Analyzer's three named extension contracts (`IAnalyticsEventStateProvider`, `IVisitorIdentifier`, `BaseVisitorIdentifier`); cascade-step impl, receipt entity, and composer are excluded.
- **No new outbox event types in slice 002**: the discipline of routing async cross-boundary work through Customizer's dispatcher (Principle VIII / D7) is established here but only materialises when slice 003+ introduces session-lifecycle webhooks. Slice 002 itself emits no webhooks; the dispatcher dependency is not touched yet.
- **Migration discoverability**: Umbraco's migration pipeline already runs Customizer's migrations on first boot; Analyzer's `IMigrationPlan` is composed alongside (per slice 001's composer ordering — `[ComposeAfter(typeof(VisitorAnalyticsComposer))]`), so Analyzer's migrations run after Customizer's. No new migration framework is introduced.
- **The reference-doc parity citations are scope-aligned**: `FR-COL-01`, `FR-COL-04`, `FR-IDP-04`, `FR-ID-05` are all in the spec's in-scope `FR-*` set (per Constitution Principle II); none of the dropped prefixes (`FR-DEP-*`, `FR-DIM-04`, `FR-DIM-03`, §3.3 bot detection, §6.2 cookie consent) appear in this slice.
- **Retention / pruning is deferred** (Clarifications Q1): slice 002 ships no retention policy and no pruning job. Receipts accumulate at the rate of pageview capture. The `IDX_ReceivedUtc` index is the only retention-related affordance shipped, enabling a later slice's date-range pruning scan to remain efficient without a table scan. Constitution Principle VII's "configurable retention" requirement is acknowledged as a forward obligation and will land alongside the broader anonymisation cascade work (anticipated: slice 013) when sessions, custom events, video, forms, scroll, and search tables also exist and can be pruned by a single cross-cutting policy.
