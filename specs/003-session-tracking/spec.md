# Feature Specification: Sessions

**Feature Branch**: `003-session-tracking`

**Created**: 2026-05-18

**Status**: Draft

**Input**: User description: "Slice 003 — Sessions. Analyzer owns the session concept (intranet contract §4): a bounded sequence of pageviews by one visitor on one device within a configurable inactivity timeout. Builds on slice 002's analyzerEventReceipt (each receipt belongs to exactly one session) and slice 002's IAnalyticsEventStateProvider (extend with CurrentSession). Introduces analyzerSession table, session-resolution at PageviewCaptured time, inactivity-timeout sweeper, additive M0002 migration adding sessionKey FK to receipts, cascade-step that deletes/anonymises a visitor's sessions in the outer scope, configurable Analyzer:Session:InactivityTimeoutMinutes (default 30)."

## User Scenarios & Testing *(mandatory)*

### User Story 1 — Sessions are created and continued automatically as visitors browse (Priority: P1)

When an authenticated EntraID employee renders pages on the host, Analyzer's session resolver — triggered by Customizer's `PageviewCaptured` notification — decides whether the captured pageview extends an existing in-progress session for that visitor+device or opens a new one. If the most-recent active session for the same `(visitorProfileKey, deviceKey)` pair has activity within the configured inactivity window, the captured pageview is attached to it (pageview count incremented, `lastActivityUtc` advanced); otherwise a new session row is opened. The slice-002 `analyzerEventReceipt` row is extended with a `sessionKey` FK so every receipt is attributable to exactly one session. `IAnalyticsEventStateProvider.CurrentSession` exposes the in-flight session reference to in-process consumers in the rare case the handler completes before the request scope ends.

**Why this priority**: Every slice from here on (custom events, video, scroll, forms, search, reports, content app) attributes events to sessions for the "visit-level engagement" metrics that operators consume — content app's "average pages per session," reports' "active sessions today," the personalization engine's "this-session segment-set evaluation." Without slice 003 those metrics are inexpressible; the receipt rows already accumulate but they cluster only by visitor + day, not by visit. Slice 003 is the minimum viable definition of a session and unblocks every later story-level analytics surface.

**Independent Test**: An authenticated EntraID employee navigates to three pages on the same browser within 5 minutes, then waits past the configured inactivity timeout, then navigates to a fourth page. Reading the `analyzerSession` table afterwards shows exactly **two** session rows for that visitor+device combination: the first with `pageviewCount = 3` and `endUtc = <last-activity-of-the-burst + timeout>`, the second with `pageviewCount = 1` and `isActive = true`. Reading `analyzerEventReceipt` shows all four receipts carrying a non-null `sessionKey`, three pointing at the first session and one at the second.

**Acceptance Scenarios**:

1. **Given** an authenticated EntraID request from a visitor with no prior active session on this device, **When** the host renders a page and `PageviewCaptured` fires, **Then** the resolver opens a new `analyzerSession` row with `startUtc = receivedUtc`, `lastActivityUtc = receivedUtc`, `pageviewCount = 1`, `isActive = true`, `endUtc = null`, and the matching `analyzerEventReceipt` row's `sessionKey` references it.
2. **Given** an authenticated EntraID request from a visitor whose most-recent active session on this device has `lastActivityUtc` within the inactivity window, **When** `PageviewCaptured` fires, **Then** the resolver attaches the new receipt to that existing session — `lastActivityUtc` advances to the new `receivedUtc`, `pageviewCount` increments by 1, and no new session row is opened.
3. **Given** an authenticated EntraID request whose most-recent active session for the same `(visitorProfileKey, deviceKey)` pair has `lastActivityUtc` older than the inactivity window, **When** `PageviewCaptured` fires, **Then** the resolver closes the stale session (sets `endUtc`, sets `isActive = false`) and opens a new session for this pageview, both updates participating in the same outer scope.
4. **Given** two concurrent `PageviewCaptured` notifications for the same `(visitorProfileKey, deviceKey)` pair (a possibility under Customizer's fire-and-forget dispatch), **When** both resolvers run on different thread-pool threads, **Then** exactly one new session row is created and both receipts attach to it — concurrency-safe session opening.
5. **Given** a consumer resolves `IAnalyticsEventStateProvider` inside a request scope where Analyzer's handler has completed before the request thread, **When** the consumer reads `CurrentSession`, **Then** the in-flight session record is returned (or `null` if the handler has not yet completed — typical for the pageview request itself, reliable for slice 004's in-request dispatches).
6. **Given** the slice-002 `analyzerEventReceipt` table already has rows when slice 003 deploys, **When** the additive migration `M0002` runs, **Then** the table gains a nullable `sessionKey` column (existing rows have `sessionKey = null` and are NOT back-filled) and the column is constrained as a soft FK (indexed, no FK constraint) to `analyzerSession.sessionKey`.

### User Story 2 — Operator-triggered visitor anonymisation soft-anonymises the visitor's sessions (Priority: P2)

When an authorised backoffice user invokes Customizer's "anonymise visitor" operator action, Customizer's `AnonymizeVisitorProfileHandler` runs every registered `IAnonymizationCascadeStep` inside its outer NPoco scope. Analyzer registers a session-cascade step that **soft-anonymises** the visitor's session rows — setting `anonymizedUtc = now`, blanking the `deviceKey` column (so the post-anonymisation rows carry no identifying device dimension), and leaving the rest of the session record intact (start/end/duration/pageview-count). This is a deliberate divergence from slice 002's receipt cascade-step (which hard-deletes) — sessions carry aggregate value (visit counts per content node, average pages per session over the rolling 90-day window) that is lost forever if rows are deleted. Constitution Principle IV v1.1.1 explicitly authorises this per-table choice between delete / soft-delete / re-projection. The slice-002 receipt rows continue to be hard-deleted (their value is per-row attribution, not aggregate); the cross-cutting analytics rollups remain accurate because they project from sessions, not receipts.

**Why this priority**: Constitution Principle IV mandates that every Analyzer-owned table register a cascade step the moment the table is introduced. Slice 003 introduces `analyzerSession`, so the cascade step must ship with it; absence is a Constitution Check failure. P2 because the trigger (operator anonymisation) is a lower-frequency event than session resolution (P1 happens on every pageview).

**Independent Test**: Seed two visitor profiles A and B, drive a small set of pageviews against each so both have sessions with `pageviewCount > 1`, then invoke Customizer's `AnonymizeVisitorProfileCommand` for visitor A. Re-read `analyzerSession`: visitor A's rows have `anonymizedUtc` populated, `deviceKey` blank, but `pageviewCount`, `startUtc`, `endUtc` preserved; visitor B's rows are unchanged. The `customizerVisitorProfile` row for A is now anonymised; the `analyzerEventReceipt` rows for A are gone (slice-002 cascade); A's sessions are still queryable in aggregate.

**Acceptance Scenarios**:

1. **Given** `analyzerSession` rows exist for two visitor profiles A and B, **When** an authorised operator invokes Customizer's anonymise action for A, **Then** Analyzer's session-cascade step sets `anonymizedUtc = now` and clears `deviceKey` on every row referencing A, leaves B's rows untouched, and the work completes inside Customizer's outer scope.
2. **Given** Analyzer's session-cascade step throws unexpectedly mid-run, **When** the outer `AnonymizeVisitorProfileHandler` disposes its scope without calling `Complete()`, **Then** the entire anonymisation rolls back atomically — the partial-update sessions revert to their pre-anonymisation state, Customizer's visitor row is not overwritten, and slice-002's receipt-deletion (which also ran inside the same outer scope) is also undone.
3. **Given** the session-cascade step runs against a visitor with zero `analyzerSession` rows, **When** the step executes, **Then** it returns successfully with no rows updated and emits no warning.
4. **Given** a session that was already anonymised on a prior cascade run (idempotent re-invocation), **When** the step runs again, **Then** the row remains anonymised — `anonymizedUtc` may advance to the new operation time or stay stable (either is acceptable; both leave the row in the same logical state).

### User Story 3 — Inactive sessions are closed automatically by a background sweeper (Priority: P3)

A configurable background service runs on the Umbraco host and, at a regular cadence, scans `analyzerSession` for rows where `isActive = true` AND `lastActivityUtc + inactivityTimeout < now`. For each match it sets `isActive = false`, sets `endUtc = lastActivityUtc + inactivityTimeout` (the timestamp the session logically ended, not the sweeper's wall-clock time), and writes the rows in bounded batches. The sweeper is the safety net for sessions whose visitors never returned to trigger the lazy-close branch in US1 AS3 — a visitor reading a single page and never coming back leaves an open session forever without the sweeper.

**Why this priority**: P3 because the lazy-close path in US1 AS3 covers every session that has at least one follow-up pageview, which is the load-bearing common case. The sweeper is the long-tail correctness mechanism that ensures session-level reports' `isActive` filter behaves correctly for visitors who don't return. Functionality is incremental on top of US1 — without it, single-pageview sessions stay open until manual intervention.

**Independent Test**: With the inactivity timeout set to a small value (e.g., 60 seconds) for the test, capture one pageview for a visitor, wait `timeout + sweepInterval + small buffer`, then read `analyzerSession`: the row's `isActive = false` and `endUtc = startUtc + timeout` (NOT the sweeper's run time, which would be later).

**Acceptance Scenarios**:

1. **Given** an `analyzerSession` row exists with `isActive = true` and `lastActivityUtc + inactivityTimeout < now`, **When** the sweeper tick runs, **Then** the row is updated to `isActive = false`, `endUtc = lastActivityUtc + inactivityTimeout`.
2. **Given** a session whose `lastActivityUtc + inactivityTimeout > now` (still within the active window), **When** the sweeper runs, **Then** the row is left unchanged — sweeper never closes an actively-extending session.
3. **Given** a session already closed by the US1 AS3 lazy-close path (`isActive = false` already), **When** the sweeper visits the row, **Then** the row is left unchanged — sweeper is idempotent against already-closed sessions.
4. **Given** the sweeper is processing a batch when the host shuts down, **When** shutdown completes, **Then** any sessions not yet processed remain `isActive = true` with their original `lastActivityUtc` — the next host's first sweeper tick picks them up. No data loss, no double-close.

### Edge Cases

- **Concurrent pageviews for the same visitor+device opening duplicate sessions**: addressed by US1 AS4 — the session resolver MUST guard against the race by using an upsert-with-uniqueness pattern (e.g., insert with a partial unique index on `(visitorProfileKey, deviceKey)` filtered to `isActive = true`, then catch-and-retry-read on collision). Exactly-one-active-session per `(visitor, device)` is the load-bearing invariant.
- **deviceKey unstable across browser sessions**: deviceKey is derived from a hash of the request's User-Agent header (per Assumptions). A visitor who upgrades their browser, clears their cookies (no cookie surface in Analyzer anyway), or switches between Chrome and Edge on the same device gets a new deviceKey and therefore a new session — by design. Operators reading session reports understand "device" as "browser instance," not "physical machine."
- **Anonymous request slipping through**: cannot happen — Customizer's middleware only publishes `PageviewCaptured` for resolved `VisitorProfileKey`s; slice 002's handler skips notifications with `VisitorProfileKey = Guid.Empty`. Slice 003's resolver inherits this skip.
- **Inactivity timeout changed at runtime**: in-flight sessions continue against the timeout active at their next pageview-resolution moment. There is no migration of existing session boundaries against the old timeout. The sweeper picks up the new value via `IOptionsMonitor<T>` at its next tick.
- **Customizer drops the parent `customizerPageview` row under back-pressure**: the receipt's `sessionKey` still points at a valid session — sessions exist independently of the parent pageview row. The cross-table join `analyzerEventReceipt JOIN customizerPageview` loses these rows by design (per slice 002's soft-FK contract); the `analyzerEventReceipt JOIN analyzerSession` join remains intact.
- **PageviewCaptured fires but the receipt's queue-write is dropped (slice 002 back-pressure)**: a sessionKey is computed for the receipt nonetheless because session resolution happens BEFORE the receipt enqueue. If the enqueue is dropped, the session still records the pageview by incrementing `pageviewCount` and advancing `lastActivityUtc`. **Net effect**: `pageviewCount` on the session can exceed `COUNT(*) FROM analyzerEventReceipt WHERE sessionKey = …` under back-pressure. Operators querying session-level aggregates see the truth; receipt-level joins see only the persisted subset. This is a documented divergence, not a defect.
- **Anonymisation cascade arrives mid-session-extension**: Customizer's orchestrator serialises against the visitor's write queue (slice-007 contract); Analyzer's session resolver inherits that serialisation. No Analyzer-side locking needed.
- **Sweeper batch exceeds DB statement timeout**: the sweeper paginates with a configurable batch size (default 1000 rows per tick) so individual update batches stay well below any reasonable SQL timeout.
- **Multiple Umbraco instances behind a load balancer running the sweeper**: each instance's sweeper does the same query; the row-level update is idempotent (setting `isActive = false` twice is a no-op). No distributed lock required. (Aggregate-level race: two sweepers may compute slightly different `endUtc` if they read the row mid-update — they converge after one full tick.)

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: Analyzer MUST extend its slice-002 `PageviewCapturedHandler` to run session resolution before enqueueing the event-receipt write op. The resolved `sessionKey` MUST be carried on the receipt write op so the receipt row that lands in the dispatcher batch carries the FK.
- **FR-002**: The session resolver MUST, for each `(visitorProfileKey, deviceKey)` pair: (a) read the most-recent `analyzerSession` row WHERE `visitorProfileKey = @visitorKey AND deviceKey = @deviceKey AND isActive = true`; (b) if found AND `lastActivityUtc + inactivityTimeout >= receivedUtc`, attach to it — UPDATE `lastActivityUtc = receivedUtc, pageviewCount = pageviewCount + 1`; (c) if found but stale, close it (`isActive = false, endUtc = lastActivityUtc + inactivityTimeout`) AND open a new session; (d) if not found, open a new session. All session-side writes MUST happen synchronously to the handler thread (not via Analyzer's bounded write queue) because the receipt's `sessionKey` FK depends on the row being durable.
- **FR-003**: Analyzer MUST introduce the `analyzerSession` table with columns: `id` (Guid PK, opaque), `sessionKey` (Guid; alternate stable identifier exposed on the public surface), `visitorProfileKey` (Guid FK to `customizerVisitorProfile.Key`), `deviceKey` (NVARCHAR(64) — see Assumptions), `startUtc` (DateTimeOffset), `lastActivityUtc` (DateTimeOffset), `endUtc` (DateTimeOffset NULL), `pageviewCount` (int, default 1), `isActive` (bit), `anonymizedUtc` (DateTimeOffset NULL). Indexes: unique non-clustered on `sessionKey`; partial unique non-clustered on `(visitorProfileKey, deviceKey) WHERE isActive = 1` (enforces "exactly one active session per visitor+device"); non-clustered on `(isActive, lastActivityUtc)` for the sweeper's predicate; non-clustered on `visitorProfileKey` for the cascade step's UPDATE-by-key. Hard FK to `customizerVisitorProfile(key)`. No FK to `analyzerEventReceipt` (the soft reverse FK from receipt → session lives on the receipt table).
- **FR-004**: Analyzer MUST ship migration `M0002_AddAnalyzerSessionTableAndReceiptSessionKey` — an additive, idempotent `AsyncMigrationBase` migration that (a) creates `analyzerSession` per FR-003, (b) adds a nullable `sessionKey` column to `analyzerEventReceipt` with a non-clustered index `IDX_analyzerEventReceipt_sessionKey`. Existing slice-002 receipts MUST NOT be back-filled — they keep `sessionKey = null` as a documented "pre-sessions" cohort. Re-runnable safely via `TableExists`/`ColumnExists` guards.
- **FR-005**: Analyzer MUST extend the public `IAnalyticsEventStateProvider` (slice-002 contract) with a new read-only member `CurrentSession` returning an immutable `AnalyticsSession` record (or `null` when the handler has not yet completed for this request). The new member is purely additive — slice 002's `CurrentRequestReceipt` member MUST remain unchanged in name, return type, and semantics. The pinning baseline MUST be regenerated to capture both members.
- **FR-006**: Analyzer MUST register a second `IAnonymizationCascadeStep` (`AnalyzerSessionCascadeStep`) alongside slice-002's `AnalyzerEventReceiptCascadeStep`. The new step MUST execute inside Customizer's outer NPoco scope BEFORE the visitor-row overwrite. The step MUST **soft-anonymise** sessions for the supplied `visitorProfileKey`: set `anonymizedUtc = now`, clear `deviceKey` (set to empty string), leave all other columns intact. (Distinct from slice-002 receipts which hard-delete — Principle IV v1.1.1 authorises the per-table choice.) Step ordering relative to slice-002 cascade step is irrelevant because both operate on disjoint tables and both participate in the same outer transaction.
- **FR-007**: Analyzer MUST register `AnalyzerSessionSweeperService : BackgroundService` (singleton hosted service) that on a configurable cadence (default 60 seconds, configurable via `Analyzer:Session:SweepIntervalSeconds`) scans for `analyzerSession` rows where `isActive = true AND lastActivityUtc + inactivityTimeout < now`. For each match it sets `isActive = false, endUtc = lastActivityUtc + inactivityTimeout` (NOT `now` — the logical close-time is when the inactivity window expired, not when the sweeper observed it). Updates are batched (default batch size 1000, configurable via `Analyzer:Session:SweepBatchSize`). The sweeper MUST swallow per-tick exceptions and log them at error level without exiting the loop, matching slice-002's `AnalyzerEventReceiptWriteDispatcher` resilience pattern.
- **FR-008**: Configuration via `IOptions<AnalyzerSessionOptions>` bound from `Analyzer:Session`: `InactivityTimeoutMinutes` (default 30), `SweepIntervalSeconds` (default 60), `SweepBatchSize` (default 1000), `CacheCapacity` (default 10 000 — see FR-010). `InactivityTimeoutMinutes`, `SweepIntervalSeconds`, and `SweepBatchSize` MUST be reloadable at runtime via `IOptionsMonitor<T>` (no host restart required to retune). `CacheCapacity` is captured at first `AnalyzerSessionCacheStore` construction and a change requires host restart — the underlying `MemoryCache.SizeLimit` is bound for the lifetime of the cache (acceptable operational tradeoff; documented in [`data-model.md`](data-model.md) §7).
- **FR-009**: The session-resolution path MUST be observably side-effect-light. Per-request budget: ≤ 1 indexed SELECT + ≤ 1 INDEXED UPDATE (extend) OR ≤ 1 INSERT (open new). Lazy-close branch adds ≤ 1 additional UPDATE (close stale). No N+1 queries; no synchronous I/O beyond these reads/writes.
- **FR-010**: The slice MUST NOT regress slice-002's published throughput envelope (1000 pv/s sustained, 5000 pv/s peak) by more than **3 ms p95 request-thread latency delta** vs slice-002 baseline. The synchronous session-row read+update on the handler thread is the load-bearing cost; the resolver MUST cache the most-recent-active-session-per-visitor lookup in an in-memory bounded LRU (matches Customizer's `IVisitorProfileStore` pattern) sized to `Analyzer:Session:CacheCapacity`.
- **FR-011**: Analyzer MUST publish a new public record type `Analyzer.Analytics.AnalyticsSession` exposing `(SessionKey: Guid, VisitorProfileKey: Guid, StartUtc: DateTimeOffset, LastActivityUtc: DateTimeOffset, EndUtc: DateTimeOffset?, PageviewCount: int, IsActive: bool)` — the consumer-facing projection. The `deviceKey` column is intentionally NOT exposed on the public record (it's a server-side resolution artefact; downstream consumers attributing sessions to devices should derive from the receipt's User-Agent when a future slice surfaces that column). The record MUST live in the `Analyzer.Analytics` namespace alongside `IAnalyticsEventStateProvider` and `AnalyticsEventReceipt` so the pinning baseline captures it directly.
- **FR-012**: Cited reference-doc parity items: `FR-EVT-01` (sessions are bounded sequences of interactions by a single visitor on a single device within a configurable inactivity timeout — satisfied by FR-002 + FR-008), `FR-EVT-02` (a session ends when inactivity exceeds the timeout — satisfied by FR-007 sweeper + FR-002 lazy-close), `FR-EVT-03` (every event in a session must be attributable to the same visitor and device — satisfied by the `(visitorProfileKey, deviceKey)` keying invariant + the `sessionKey` FK on every receipt), `FR-COL-01` (server-side capture continues — inherited from slice 002 with no regression), `FR-COL-04` (no third-party data store — sessions live in the same `IUmbracoDatabase`). None of the dropped prefixes are cited.
- **FR-013**: All slice-003 public types — `IAnalyticsEventStateProvider` (modified — `CurrentSession` added), `AnalyticsSession` (new), plus the slice-001 + slice-002 types already pinned (`AnalyticsEventReceipt`, `IVisitorIdentifier`, `VisitorIdentity`) — MUST be captured by `PublicSurfacePinningTests`. The slice-003 baseline regeneration MUST be documented with a Sync Impact-style note in this spec (or its successor amendment) explaining the new public member.

### Key Entities

- **Session (Analyzer-owned, new this slice)**: A bounded sequence of pageviews by one visitor on one device within the inactivity timeout. Keyed by an opaque `id` PK + a publicly-exposed stable `sessionKey`. Hard FK to `customizerVisitorProfile.Key`. Carries the activity window (`startUtc` / `lastActivityUtc` / `endUtc`), pageview count, active flag, anonymisation marker. The "device" dimension is internal (see Assumptions) — public consumers don't see `deviceKey`.
- **Analytics-event receipt (Analyzer-owned, modified this slice)**: The slice-002 `analyzerEventReceipt` table gains a nullable `sessionKey` column — soft FK to `analyzerSession.sessionKey` (indexed, no DB-level constraint because the session row is opened synchronously to the handler thread and is durable; the soft-FK affordance is for forward-compatibility with future cases where a receipt may legitimately have no session). Pre-existing slice-002 rows keep `sessionKey = null` (no back-fill).
- **Visitor profile (read-only at this slice, owned by Customizer)**: Sessions FK to `customizerVisitorProfile.Key`. Never inspected for its `IdentityRef`.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: For an authenticated EntraID employee browsing **N consecutive pages within the inactivity window**, exactly **one** `analyzerSession` row exists with `pageviewCount = N` and `isActive = true` immediately after the Nth page renders. (100% of test cases covering N=1, N=3, N=10, N=50.)
- **SC-002**: For an authenticated EntraID employee browsing a page, waiting `inactivityTimeout + 1 minute`, then browsing another page, exactly **two** `analyzerSession` rows exist for that visitor+device combination — the first with `isActive = false, endUtc = startUtc + inactivityTimeout`, the second with `isActive = true, pageviewCount = 1`.
- **SC-003**: Under sustained load matching slice-002's published envelope (1000 pv/s for 60 seconds), the 95th-percentile request-thread latency delta vs the slice-002-only baseline is **≤ 3 ms**, and no session-row write is observed to block the request thread for more than **10 ms** (sub-budget for the synchronous extend/open writes per FR-009).
- **SC-004**: Operator-triggered visitor anonymisation soft-anonymises **all** of the visitor's `analyzerSession` rows in **≤ 200 ms for a visitor carrying ≤ 1 000 sessions** (matching slice-002's SC-003 budget shape). The post-anonymisation state preserves `pageviewCount`, `startUtc`, `endUtc`; clears `deviceKey`; sets `anonymizedUtc`.
- **SC-005**: The background sweeper closes 100% of eligible sessions (rows where `isActive = true AND lastActivityUtc + inactivityTimeout < now`) within `2 × SweepIntervalSeconds` of becoming eligible. (Sweeper convergence under steady state.)
- **SC-006**: Concurrent `PageviewCaptured` notifications for the same `(visitorProfileKey, deviceKey)` pair produce exactly **one** new session row (race-safety) in 100% of test cases covering up to 10 concurrent dispatches.
- **SC-007**: The public-surface pinning baseline regenerates without diff against the checked-in baseline. The baseline includes `IAnalyticsEventStateProvider` with both `CurrentRequestReceipt` (slice 002) and `CurrentSession` (slice 003) members, plus `AnalyticsSession` as a new public record.
- **SC-008**: Slice 003 ships with passing unit and integration tests covering all US1 acceptance scenarios (1–6), all US2 acceptance scenarios (1–4), and all US3 acceptance scenarios (1–4). Integration tests run against a real SQL Server container (matching slice 002's pattern).

## Assumptions

- **deviceKey derivation**: deviceKey is a stable server-side identifier derived from `notification.Pageview.UserAgent` — concretely, a truncated SHA-256 hash (16 hex characters) of the normalised User-Agent string. **Source**: the raw UA is captured synchronously on the request thread by Customizer's `PageviewCaptureMiddleware` (cross-product prerequisite — see [`customizer-prereq.md`](customizer-prereq.md)) and threaded through the `PageviewCaptured` notification as the 10th positional record param (`Pageview.UserAgent`). Analyzer's handler reads it from the notification at handler-entry time; this is reliable regardless of whether the handler runs on the request thread or on a post-request `Task.Run` thread, because the UA value is part of the immutable `Pageview` record. **Rationale**: no new cookie surface (consistent with the product framing — "no cookie-consent / opt-out surface"); no new persistence (Analyzer hashes UA to deviceKey before its own write; Customizer keeps the raw UA in-memory only); reproducible across requests; bounded cardinality (≤ a few hundred distinct UA strings per organisation in practice). **Tradeoff**: a visitor on Chrome and Edge on the same physical laptop is two devices, not one — acceptable for analytics framing where "device" means "browser instance." When a future slice surfaces a richer device dimension, this assumption may be revisited.
- **Cascade-step semantic is soft-anonymise** (Principle IV v1.1.1 per-table choice): sessions are soft-anonymised on operator-triggered anonymisation rather than hard-deleted (the slice-002 receipt cascade-step's choice). **Rationale**: session-level rollups (visit counts per content node, average pages per session over rolling windows) are load-bearing for slice 005's content app and slice 010's reports, and a single visitor's anonymisation would create artificial dips in those aggregates if rows were deleted. Soft-anonymise preserves aggregate counts; blanking `deviceKey` and setting `anonymizedUtc` keeps the row CCPA-defensible (no PII remains on the session row).
- **Session row writes are synchronous to the handler thread** (not via the slice-002 bounded write queue): the receipt's `sessionKey` FK depends on the session row being durable at the moment the receipt write op enqueues. Going through the queue would race the receipt write against the session insert. The per-pageview cost (≤ 1 indexed SELECT + 1 INSERT or UPDATE) is bounded by the in-memory most-recent-active-session cache (FR-010); cache misses fall to the SQL Server, which under 1000 pv/s steady state at ~10 000 distinct visitors is a small fraction of pageviews.
- **In-memory active-session cache** (FR-010): an LRU cache holds `(visitorProfileKey, deviceKey) → most-recent active sessionKey + lastActivityUtc` for ≤ 10 000 entries (configurable). Cache invalidation happens on session close (`isActive = false`) and on session anonymisation. Cross-instance cache coherence is **not** maintained — two Umbraco instances behind a load balancer may briefly disagree on the active session for a visitor, resolved by the partial unique index on `(visitorProfileKey, deviceKey) WHERE isActive = 1` (the second instance's INSERT collides and falls back to a fresh-read-then-extend).
- **Sweeper close-time semantic**: when the sweeper closes a session, it sets `endUtc = lastActivityUtc + inactivityTimeout` (the logical close time), NOT `now` (the sweeper's observation time). **Rationale**: session-duration metrics depend on `endUtc - startUtc`; using `now` would inflate every long-tail session's duration by up to one full sweeper interval. The logical close time is correct for analytics.
- **In-flight cache misses fall to the database synchronously**: there is no asynchronous prefetch or warm-on-boot pattern. The first pageview after host restart pays a SELECT cost; subsequent ones within the cache window don't. Acceptable per FR-010's p95 budget.
- **Sessions for already-anonymised visitors**: if a `PageviewCaptured` notification arrives for a visitor profile whose `IdentityRef` is `anonymized:…` (per Customizer's slice-007 anonymisation pattern), the handler MUST proceed normally — the session resolver doesn't inspect `IdentityRef` and the receipt's `visitorProfileKey` FK is still valid. Sessions opened for anonymised visitors are themselves treated as anonymised (`anonymizedUtc` set on open).
- **Inter-product contract reference**: §4 of `docs/INTER-PRODUCT-CONTRACT.md` is the source-of-truth for "Analyzer owns sessions." Customizer's `PageviewCaptured` (slice 011, already shipped at `05e989c`), `IAnonymizationCascadeStep` (slice 007, already shipped), AND a new additive `Pageview.UserAgent` field (see [`customizer-prereq.md`](customizer-prereq.md); contract §6 item 2; cross-product slice paired with Analyzer slice 003 per `/speckit-analyze` finding C1) are the contracts slice 003 consumes. The new `Pageview.UserAgent` is the second item added to §6 — additive to the inter-product contract; no other Customizer-side change.
- **Public-surface pinning regeneration**: slice 003 changes `IAnalyticsEventStateProvider` additively (new `CurrentSession` member) AND adds a new `AnalyticsSession` record. The baseline file at `src/Analyzer.Tests/PublicSurface/Baselines/Analyzer-public-surface.txt` is regenerated as part of this slice (via `ANALYZER_REGENERATE_SNAPSHOTS=1`) with the additive change documented in a Sync Impact-style note. The change is MINOR-additive (no breaking member rename or removal), consistent with Constitution Principle X.
- **No reports, no content app**: slice 003 ships no operator-facing surface. Sessions become queryable in SQL but no backoffice view aggregates them. Aggregation surfaces land in slice 005 (per-content-node Analytics content app) and slice 010 (reports). The session-level aggregates implied by SC-005 are testable via direct SQL only at this slice.
- **No cross-device session merging**: explicitly out of scope (per the user description). Two devices for the same visitor produce two parallel session lines; no Analyzer logic attempts to merge them.
